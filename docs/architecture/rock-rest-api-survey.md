# Rock REST API Survey for My Groups

## Overview

Surveyed the Swagger UI at `https://rock.christfellowship.church/api/docs/index` (Rock REST API v1) on 2026-07-23. The Rock CMS version number is not exposed anywhere in the Swagger UI itself — only "Rock Rest API v1" is shown. The instance exposes two parallel API surfaces relevant to this project: a classic per-controller REST API (`/api/{Controller}`, one controller per Rock entity/model, generated by an older Swashbuckle-style doc generator) and a newer generic model API (`/api/v2/models/{entity}`) that appears automatically alongside almost every classic controller. Both were surveyed together below since they overlap heavily for CRUD.

## Key findings

**On Groups & GroupMembers**

1. **List all groups a person belongs to:** No dedicated "groups by person" endpoint exists on `Groups` or `GroupMembers`. Use `GET /api/GroupMembers?$filter=PersonId eq {id}&$expand=Group,GroupRole` (OData filter/expand, confirmed supported — see #7). The `Groups` controller does have `GET /api/Groups/GetFamilies/{personId}` and `GetFamiliesByPersonNameSearch/{searchString}`, but those are family-group-specific, not general-purpose group membership.
2. **List all members of a specific group:** `GET /api/GroupMembers?$filter=GroupId eq {id}`. Pagination is supported via standard OData `$top`/`$skip` on the same endpoint (confirmed present on the base GET for every controller checked).
3. **Add a person to a group:** `POST /api/GroupMembers` with a `Rock.Model.GroupMember` body. Minimum practical payload is `GroupId`, `PersonId`, `GroupRoleId` (role is required — Rock group members always have a role, obtained from `GroupTypeRoles`).
4. **Remove a person from a group:** `DELETE /api/GroupMembers/{id}` performs a real row delete (no soft-delete parameter on the endpoint). The `GroupMember` model does carry `GroupMemberStatus` (`Inactive`/`Active`/`Pending`) and `IsArchived`/`ArchivedDateTime` fields, so a "soft remove" is possible instead by `PATCH`ing the member to `GroupMemberStatus: "Inactive"` or setting `IsArchived: true`, rather than calling DELETE. **Open question:** whether DELETE cascades to related Attendance/History records — not visible from Swagger, needs a human check against Rock's server-side behavior/docs.
5. **Promote/change a member's role:** `PATCH` or `PUT /api/GroupMembers/{id}` setting `GroupRoleId` to the target role's id (role ids come from `GET /api/GroupTypeRoles?$filter=GroupTypeId eq {id}`).
6. **Set attribute values at creation vs. separately:** The `GroupMember` model has `Attributes`/`AttributeValues` properties, but Rock's REST convention is that attribute values are **not** persisted by a generic `POST`/`PUT` on the entity — they require a follow-up call, either the v1 `POST /api/GroupMembers/AttributeValue/{attributeValueId}` (sets one attribute value by id) or the v2 `PATCH /api/v2/models/groupmembers/{id}/attributevalues` (partial update of one or more attribute values). **Flagged as best-effort**: Swagger doesn't state this explicitly; recommend a human confirm behavior before relying on it.
7. **OData support (`$filter`, `$select`, `$expand`, `$top`, `$skip`):** Yes, confirmed present as query parameters on the base `GET` (list) endpoint of every controller checked (`Groups`, `GroupMembers`, `People`, `GroupTypes`, `GroupTypeRoles`, `GroupLocations`, `Attendances`, `Schedules`, `AttributeValues`, `Attributes`, `Workflows`). Also `loadAttributes` and `attributeKeys` params exist on the list/single-item GET endpoints specifically to inline attribute values into the response without a second call.

**On People**

8. **Currently authenticated person:** `GET /api/People/GetCurrentPerson`. Related: `GET /api/People/GetCurrentPersonImpersonationToken` (creates a token for the logged-in person, used by external apps that need to act as that person).
9. **Update person profile fields via REST:** Yes — `PUT`/`PATCH /api/People/{id}` with a `Rock.Model.Person` body (any writable property, e.g. name fields, email, phone via `PhoneNumbers`, etc.). Dedicated helpers also exist: `POST /api/People/UpdateProfilePhoto` and `UpdatePersonProfilePhoto` (photo upload, body is a raw string/base64), `POST /api/People/ConfigureTextToGive/{personId}`.
10. **Update person attribute values via REST:** Same pattern as GroupMember (#6) — `GET`/`PATCH /api/v2/models/people/{id}/attributevalues`. No dedicated v1 single-attribute endpoint was found on `People` (only the v2 route).

**On Workflows**

11. **Trigger a workflow via REST:** Two mechanisms exist. (a) Every entity controller exposes a generic `POST /api/{Controller}/LaunchWorkflow/{id}` — pass an entity id (or `0` for none) and a body identifying the workflow type; this launches a workflow with that entity as the workflow's target. This matches the classic `LaunchWorkflow/0?workflowTypeId=` pattern used elsewhere in this org's integrations. (b) `POST /api/Workflows/WorkflowEntry/{workflowTypeId}` — a dedicated "initiate a new workflow" endpoint scoped to the `Workflows` controller itself.
12. **Are workflows auto-triggered by POSTing to GroupMembers, etc.?** Not answerable from Swagger alone. Rock does have a `WorkflowTriggers`/`GroupMemberWorkflowTriggers` concept in its data model (configured server-side, not visible as REST behavior in the spec), so it's plausible that some GroupMember changes fire workflows automatically depending on system configuration — but this cannot be confirmed from the API docs. **Flagged as open question** — needs a human to check Rock admin workflow trigger configuration for GroupMember entity type.

**On Attributes**

13. **How custom attribute values are read/written:** Two ways. Inline: `loadAttributes`/`attributeKeys` query params on an entity's `GET` (list or single) return `Attributes`/`AttributeValues` embedded in the response. Separately: the dedicated `AttributeValues` controller offers standard CRUD (`GET`/`POST`/`PUT`/`DELETE`/`PATCH /api/AttributeValues[/{id}]`) plus the v2 `.../attributevalues` sub-resource on each entity's v2 route. The `Attributes` controller itself is for managing attribute **definitions** (metadata), not values — its only non-boilerplate endpoints are `PUT /api/attributes/flush` and `PUT /api/attributes/flush/{id}` (cache invalidation).

**On Auth**

14. **Auth model from the REST API's perspective:** `POST /api/Auth/Login` returns a cookie ("AuthCookie") used on subsequent REST calls — so at minimum Rock supports cookie-based session auth. Separately, there are `AuthClients`, `AuthClaims`, and `AuthScopes` controllers, which are CRUD controllers over what looks like OAuth client/claim/scope configuration data — suggesting Rock also supports a token-based (OAuth-style) auth flow for REST, but the docs only expose these as data-model CRUD, not the actual token-issuance flow. **Flagged as open question**: the real authenticated-request mechanism for a SPA/mobile client (header format, token lifetime, refresh) isn't visible in Swagger and should be confirmed against Rock's REST authentication documentation or existing `remix-web-app` code (if any authenticated calls already exist there).

## Open questions

- Does `DELETE /api/GroupMembers/{id}` cascade to Attendance/History records, or leave orphaned data? (#4)
- Does setting `AttributeValues` in the body of `POST`/`PUT /api/GroupMembers` or `/api/People` actually persist, or is a separate attribute-value call always required? (#6, #10)
- Are any workflows auto-triggered server-side by entity CRUD operations (e.g. adding a GroupMember), independent of the explicit `LaunchWorkflow` call? (#12)
- What is the actual token/cookie format and lifetime for authenticated REST calls, and does this org use cookie auth, OAuth, or an API key header in practice? (#14) — the `AuthClients`/`AuthClaims`/`AuthScopes` controllers hint at OAuth support but don't document the flow.
- Rock version running on this instance is not shown anywhere in Swagger; useful to know when checking Rock's official REST docs/changelog for version-specific behavior.

## Endpoint catalog

Note on format: nearly every entity controller in this Rock instance exposes an identical boilerplate set of endpoints in addition to standard CRUD. To avoid repetition, that boilerplate is documented once here and only *deviations or additions* are listed per controller below.

**Standard CRUD (present on every controller in this survey):**
```
GET    /api/{Controller}                    Queryable GET (list) — supports $expand, $filter, $select, $orderby, $top, $skip, loadAttributes, attributeKeys
POST   /api/{Controller}                    Add a record — body: full model object
GET    /api/{Controller}/{id}               Get a single record — supports loadAttributes, attributeKeys
PUT    /api/{Controller}/{id}                Update a record — body: full model object
DELETE /api/{Controller}/{id}                Delete the record (hard delete)
PATCH  /api/{Controller}/{id}                Partial update — body: partial object
POST   /api/{Controller}/AttributeValue/{id} Set a single attribute value (id = attribute value id)
DELETE /api/{Controller}/AttributeValue/{id} Delete a single attribute value
```

**Generic "utility" endpoints (present on almost every controller):**
```
GET    /api/{Controller}/DataView/{id}                          List of records represented by a Data View
GET    /api/{Controller}/InDataView/{dataViewId}/{entityId}      Whether entityId is in the given Data View
GET    /api/{Controller}/FollowedItems                           Items the current user follows (where applicable)
GET    /api/{Controller}/GetByAttributeValue                     Filter records by an attribute value
POST   /api/{Controller}/LaunchWorkflow/{id}                     Launch a workflow against this entity (id, or 0)
PUT    /api/{Controller}/SetContext/{id}                         Set the Rock "context cookie" to this record
```

**v2 model API (present alongside almost every classic controller, lowercase plural entity name):**
```
POST   /api/v2/models/{entity}                          Create
GET    /api/v2/models/{entity}/{id}                     Get single
PUT    /api/v2/models/{entity}/{id}                      Full update
DELETE /api/v2/models/{entity}/{id}                      Delete (not present on People)
PATCH  /api/v2/models/{entity}/{id}                      Partial update
GET    /api/v2/models/{entity}/{id}/attributevalues      Get all attribute values
PATCH  /api/v2/models/{entity}/{id}/attributevalues      Partial update of attribute values
POST   /api/v2/models/{entity}/search                    Search by user-defined query (body: EntitySearchQueryBag)
GET/POST /api/v2/models/{entity}/search/{searchKey}      Search by a named system query
```

### Tier 3: Noted for reference

- **GroupHistoricals / GroupLocationHistoricals / GroupMemberHistoricals** — historical/point-in-time snapshots of group, group-location, and group-member data (for reporting/trend analysis).
- **GroupMemberAssignments** — assignment records for a group member (e.g. scheduling assignments).
- **GroupMemberRequirements / GroupRequirements / GroupRequirementTypes** — tracks and defines requirements (background checks, trainings) a person must meet to be a group member.
- **GroupMemberScheduleTemplates / GroupScheduleExclusions / PersonScheduleExclusions / CampusSchedules** — scheduling templates and exclusion dates for group/attendance scheduling.
- **GroupMemberWorkflowTriggers / GroupSyncs** — server-configured triggers that fire workflows or sync group membership from a data view; standard CRUD entity controllers, actual trigger logic runs server-side and isn't visible via REST.
- **GroupDemographicTypes / GroupDemographicValues** — custom demographic classification fields for groups.
- **Communications / CommunicationRecipients / CommunicationResponses / CommunicationTemplates (+Attachments variants)** — Rock's messaging/email-blast system; relevant if My Groups leaders need to message members.
- **Auth / Auths / AuthAuditLogs / AuthClaims / AuthClients / AuthScopes** — login (`POST /api/Auth/Login` returns a cookie) plus CRUD over OAuth-style client/claim/scope records. See Key Finding #14 and Open Questions.
- **UserLogins / HistoryLogins / RemoteAuthenticationSessions / IdentityVerifications / IdentityVerificationCodes / PersonTokens** — supporting login/identity-verification/session infrastructure; standard CRUD, worth a closer look only if building custom login rather than using existing Rock login.
- **PersonSearchKeys / PersonDuplicates / PersonPreviousNames** — search-key and duplicate-detection data for the People module.
- **PhoneNumbers** — standard CRUD for a person's phone numbers (separate from the `Person` model's own fields).
- **Notes / NoteTypes / NoteAttachments / NoteWatches** — Rock's generic notes system, could be used for leader notes on a group or member.
- **Reminders / ReminderTypes** — Rock's reminder system, could support "remind me to follow up with this member."
- **Tags / TaggedItems** — generic tagging system, could be used for group/member categorization beyond Group Types.

## Notes and observations

- **Two parallel APIs exist.** Nearly every classic `/api/{Controller}` controller has a matching `/api/v2/models/{entity}` route with its own CRUD + search + attribute-values sub-resource. The v2 routes are more consistent (e.g. always have a dedicated `attributevalues` sub-resource) and use a query-object-based `search` endpoint (`EntitySearchQueryBag`) instead of raw OData strings. Worth deciding up front whether My Groups standardizes on v1 (OData, more mature/documented) or v2 (newer, more uniform) rather than mixing both.
- **Surprise:** No entity controller offers a first-class "list related child records" shortcut in the general case (e.g. no `GET /api/People/{id}/Groups`). Nearly everything relational is done through `$filter` on the child controller (e.g. `GroupMembers?$filter=PersonId eq X`) rather than a nested path. Plan client-side code around that pattern.
- **Surprise:** Attribute values are consistently treated as a second-class, separate write — no controller surveyed appeared to accept attribute values inline on create. This has real implications for a "create group member with custom fields" flow: it will likely require two API calls (create, then set attributes), not one.
- **GroupTypeRoles and AttributeValues have no v2 route**, while most other Tier 1/2 controllers do — inconsistent coverage of the newer API across entities.
- Controller naming is inconsistent in one spot: it's `PersonAlias` (singular) in the sidebar, unlike every other controller which is pluralized.
- Rock's REST auth story is the least-documented area from Swagger alone — a real answer likely requires either testing an authenticated call, or checking `remix-web-app`'s existing (public/anonymous) Rock REST usage for any existing auth precedent, or Rock's own written REST documentation outside Swagger.
